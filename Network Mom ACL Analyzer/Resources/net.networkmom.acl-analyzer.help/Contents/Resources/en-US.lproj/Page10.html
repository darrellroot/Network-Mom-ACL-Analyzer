About "do not care bits"

Network engineers are familiar with netmasks. For IPv4 they indicate the number of leading bits in the “network” portion. For example, a netmask of 255.0.0.0 corresponds to a /8 or an 8-bit netmask. Cisco ASA’s use netmasks in their ACL configuration.

Cisco IOS devices use a “do not care bit” configuration. For example, to specify the network 10.1.2.0/24 you would use “10.1.2.0 0.0.0.255”. Many network engineers call this an “inverse netmask” but that is  incorrect . The configuration is not specifying the netmask at all! It is specifying which of the 32 bits in the IPv4 address do not need to match for the rule to apply.

For example, this is a valid “do not care bit” configuration when configuring a Cisco IOS router: “10.1.0.3 0.0.255.0”. That matches 10.1.0.3, 10.1.1.3, 10.1.2.3, and so on up through 10.1.255.3.

This is also a valid “do not care bit” configuration: “10.2.8.4 0.1.3.1”. That matches many IP addresses including 10.2.8.4, 10.3.11.5, 10.2.9.4, and 10.3.10.4. This is a “discontiguous do not care bit” configuration.

The Cisco CCIE exam used to include a task to “optimize” an ACL down to a minimal number of lines. Doing this required understanding “do not care bit” configurations.


Configuring a production network with a discontiguous “do not care bit” configuration is a bad idea.   Even if you understand “don’t care bit” arithmetic, it makes your router configuration harder to maintain. Not all the engineers who work on your network will understand them. Let Cisco IOS optimize the ACL for you! Do not make things complex to remove a handful of ACL lines.

Network Mom ACL Analyzer reports “discontiguous don’t care bits” as an error  and does not analyze them further. Remove them from your ACLs.
